# 模拟面试会话页优化 Spec

- 版本：v1.0
- 日期：2026-03-23
- 页面路径：`/interview/session`
- 对应实现：
  - `apps/web/components/interview-session-room.tsx`
  - `apps/web/app/interview/session/page.tsx`

## 1. 背景

当前会话页已经具备：

1. 题目队列加载
2. 单一对话流展示
3. SSE 流式评价
4. 动态追问插入
5. 会后复盘展示

但从产品体验看，页面仍偏“研发调试台”，还不是“高压但清晰的真实面试空间”。

核心问题不在能力缺失，而在于：

1. 主任务不够聚焦：用户当前最该做的是“理解当前题并作答”，但页面同时并列展示队列、状态、复盘、调试，注意力被打散。
2. 实时反馈不够可感知：评分流式返回了，但“系统正在做什么”“为什么插入追问”“当前回答是否已被接收”都不够明确。
3. 信息层级混杂：进行中的状态、事后复盘、开发调试同时出现，打断面试心流。
4. 指标可读性弱：`平均得分`、`证据命中` 有数值，但缺少“这意味着什么”的解释和下一步引导。
5. 操作决策成本高：`刷新队列`、`生成复盘`、`结束面试` 并列，优先级不清晰，容易误操作。

## 2. 设计目标

### 2.1 业务目标

1. 提升单场模拟面试的完成率。
2. 提升用户对“动态追问”和“证据评分”的理解度。
3. 提升复盘生成率与复盘查看率。
4. 降低首次使用该页面时的认知成本。

### 2.2 体验目标

1. 让用户始终知道“当前题是什么、做到哪一步、接下来该做什么”。
2. 让 SSE 流式评价更像“面试官正在思考和反馈”，而不是一段突然出现的文本。
3. 把“进行中信息”和“事后总结信息”分层，避免抢焦点。
4. 保留现有实现能力，不引入过重的新交互范式。

### 2.3 非目标

1. 本期不重做出题编排逻辑。
2. 本期不引入音视频面试形态。
3. 本期不新增多轮复盘编辑器。
4. 本期不把所有调试信息产品化展示给普通用户。

## 3. 用户场景

### 3.1 目标用户

1. 准备前端面试的求职者
2. 已有简历、JD、项目经历，希望进行高拟真面试训练的用户

### 3.2 核心场景

1. 用户进入会话页后，迅速理解当前题并开始回答。
2. 回答提交后，用户能感知系统正在评分、检索、生成追问。
3. 当分数偏低时，用户能理解“为什么系统继续追问”。
4. 面试结束后，用户优先查看可行动的复盘结论，而不是原始技术字段。

## 4. 现状问题拆解

### 4.1 信息架构问题

1. Header 已经承载进度、题型、难度、来源、计时、刷新、复盘、结束，过于拥挤。
2. 主区顶部再次展示当前题，和聊天流中的 AI 提问存在重复。
3. 侧栏同时出现状态、题目队列、复盘结果、调试输出，阶段性内容没有分层。
4. `题目队列` 使用折叠面板，价值高但可见性偏低。

### 4.2 实时反馈问题

1. 用户提交回答后，只能看到“正在准备评分...”，缺少更强的阶段感。
2. `stage` 信息当前只在评价气泡内弱展示，没有独立的处理状态区。
3. 动态插入追问只在局部 tag 和队列高亮中体现，解释不足。

### 4.3 决策与操作问题

1. `刷新队列` 对普通用户不是高频动作，但目前和主操作同级。
2. `生成复盘` 与 `结束面试` 容易被理解成重复动作。
3. 题目完成后虽然有提示，但没有明显引导用户进入“收尾动作”。

### 4.4 内容表达问题

1. `证据命中` 对用户过于技术化，建议转为“资料佐证”或“命中资料”。
2. `source llm`、`medium` 等标签偏内部字段直出，不够产品化。
3. 对当前分数的意义、风险和建议缺少自然语言解释。

## 5. 优化原则

1. 主流程优先：当前题、用户作答、系统反馈必须占据视觉中心。
2. 阶段分层：进行中信息优先，复盘信息次之，调试信息隐藏。
3. 状态可解释：用户随时知道系统当前在“接收回答 / 检索资料 / 评分 / 生成追问 / 等待下一题”中的哪一步。
4. 文案产品化：尽量避免后端字段名直出。
5. 尽量复用现有接口：先做前端重构，再评估补字段。

## 6. 新页面结构

### 6.1 总体布局

最终态采用“沉浸式对话页 + 右侧抽屉面板”：

1. 顶部为轻量会话状态条
2. 中间为单列主面试工作区
3. 右侧不再常驻侧栏，改为抽屉式 `面试面板`

产品原则：

1. 默认页面只服务“读题、看反馈、作答”
2. 所有辅助信息统一收进抽屉
3. 页面范式从“双栏控制台”切为“单主区对话空间”

#### 6.1.1 顶部会话状态条

保留内容：

1. 返回入口
2. 页面标题
3. 当前进度：`第 x / y 题`
4. 计时器
5. 当前状态：`等待作答 / 评分中 / 追问已生成 / 已完成`
6. 抽屉入口按钮：`面试面板`

下沉或弱化内容：

1. `userId` 不在默认视图展示
2. `source` 不在 header 展示
3. `刷新队列` 不再作为常驻一级操作
4. `生成复盘` 与 `结束面试` 不再常驻并列出现在顶部主区域

#### 6.1.2 主面试工作区

默认页面只保留 3 层：

1. 当前题卡片
2. 实时对话流
3. 固定底部回答区

说明：

1. 这 3 层构成完整作答闭环
2. 主工作区应明显大于现有实现，成为页面绝对视觉中心

#### 6.1.3 右侧抽屉面板

右侧改为 slide-over 抽屉，不再占据默认页面宽度。

抽屉承载：

1. 本场进展
2. 题目地图
3. 复盘总结
4. 开发调试信息，仅开发模式可见

交互规则：

1. 默认关闭
2. 用户点击 `面试面板` 后打开
3. 全部题目完成后，可自动打开并引导到 `复盘`

### 6.2 模块定义

#### 6.2.1 当前题卡片

目标：替代现在聊天区顶部的弱标题，明确当前题上下文。

展示内容：

1. 当前题序号：`第 2 题`
2. 题目正文
3. 题型标签
4. 难度标签
5. 若为追问，展示提示：`这是基于上一题继续深挖的追问`
6. 若刚插入追问，展示原因提示：`系统检测到你的回答在证据/取舍上仍可继续展开，因此插入追问`

交互规则：

1. 当前题切换时卡片更新
2. 题目完成且无下一题时，卡片进入完成态
3. 完成态文案：`本场题目已完成，建议先生成复盘，再决定是否结束面试`
4. 下一题的主承载区域改为“当前题卡片”，而不是在评价气泡后立刻追加完整新题气泡
5. AI 评价完成后，先让用户停留在上一题的收尾状态，再通过卡片切换进入下一题

新增切题规则：

1. 当下一题已准备好时，当前题卡片进入 `next-question-ready` 状态
2. 卡片顶部显示轻量提示：
   - 普通下一题：`这一题先到这里，下一题已准备好`
   - 追问场景：`基于你刚才的回答，接下来会继续追问这个点`
3. 卡片主区域切换为下一题题干、题型、难度
4. 聊天流不再紧跟着插入一条完整“下一题”大气泡，避免与上一条 AI 评价争抢注意力
5. 若仍需保留聊天记录完整性，可只插入一条简短系统分隔消息：
   - `进入下一题`
   - `进入追问`

这样做的目的：

1. 把“评价上一题”和“开始下一题”拆成两个心智阶段
2. 让用户优先消化评价，再进入下一题
3. 让“当前正在回答哪一题”始终由页面主区域统一承载，而不是依赖聊天流滚动位置判断

#### 6.2.2 实时对话流

保留现有单流设计，但增加“消息类型可识别性”：

1. 面试官提问
2. 用户回答
3. AI 评价
4. 系统状态提示

优化点：

1. AI 评价消息增加 `评分中` 视觉态
2. `stage` 事件不要只替换正文，增加小型进度条或步骤文案
3. 对于追问插入，增加一条系统消息：
   - `已根据本轮回答插入 1 道追问，接下来会继续深挖这个点`
4. 下一题默认不再以完整题干气泡紧贴 AI 评价后出现，而是由当前题卡片承载
5. 聊天流更偏向“历史记录区”，不再承担“下一题主展示区”职责
6. 对话流宽度应明显大于现有实现，成为默认视图的主区域

消息策略调整：

1. 保留：
   - 用户回答
   - AI 评价
   - 系统状态提示
2. 弱化：
   - 下一题完整题干气泡
3. 替代方案：
   - 用卡片切换承载下一题
   - 聊天流只保留轻量过渡消息，表示流程已进入下一题

#### 6.2.3 回答区

目标：强化“作答”这个唯一主任务。

展示内容：

1. 多行输入框
2. 主按钮：`发送回答`
3. 辅助提示：`建议尽量覆盖背景、方案、取舍、结果`

新增交互建议：

1. 增加字数提示：`建议 80-300 字`
2. 提交后短暂显示 `回答已提交，正在评分`
3. 无当前题时，不只禁用输入框，还要提供明确下一步按钮：
   - 若可复盘：`生成复盘`
   - 若已复盘：`结束面试`

本期不新增但预留：

1. `要求澄清题意`
2. `跳过此题`
3. `追问面试官`

这些能力后端已经有 intent 分支，但当前 UI 暂不强推，可在二期接入快捷操作。

#### 6.2.4 面试面板抽屉

抽屉统一承载辅助信息，不再使用常驻右侧栏。

抽屉内部必须 tab 化，避免再次退化成小型 dashboard。

固定 3 个 tab：

1. `进展`
2. `题目`
3. `复盘`

设计原则：

1. 默认进行中时用户很少需要打开抽屉
2. 打开抽屉时，用户应快速找到“我现在到哪里了”“后面还有什么题”“这场面试总结了什么”
3. 不允许把所有信息直接以长列表堆在抽屉里

#### 6.2.5 进展 Tab

替代当前 `面试状态`，强调“现在发生了什么”。

推荐字段：

1. 完成进度：`1 / 6`
2. 当前会话均分
3. 最近一轮资料佐证数
4. 当前状态文案

状态文案枚举：

1. `等待你的回答`
2. `系统正在评分`
3. `已生成评价，等待下一题`
4. `已完成全部题目`
5. `正在生成复盘`

说明：

1. `证据命中` 重命名为 `资料佐证`
2. `retrieval_strategy` 不直接展示英文内部值，改成中文说明，例如：
   - `优先查看简历/JD`
   - `优先查看知识库资料`
   - `等待评分完成`

补充规则：

1. `同步题目状态` 按钮若保留，放在该 tab 内，而不是顶栏常驻

#### 6.2.6 题目 Tab

替代当前 `题目队列` 折叠组件，默认展开。

展示目标：

1. 让用户知道整场还有多少题
2. 明确当前题位置
3. 区分普通题、追问题、已完成题、未开始题

展示规则：

1. 每题只展示一行摘要，避免过长
2. 状态用中文产品文案替换原始 status
3. 当前题高亮
4. 新插入追问用特殊标识
5. 支持点击查看完整题干，但本期不支持跳题

状态映射建议：

1. `pending` -> `待回答`
2. `answered` -> `已完成`
3. `skipped` -> `已跳过`
4. `follow_up` 额外加标签 `追问`

形式要求：

1. 使用紧凑列表，不使用大卡片堆叠
2. 当前题高亮必须明显
3. 新插入追问在列表中要有单独视觉标识

#### 6.2.7 复盘 Tab

目标：把现在“字段展示”升级成“可执行反馈”。

默认空态：

1. 标题：`面试复盘`
2. 文案：`完成几轮回答后，可生成本场复盘，沉淀优势、弱项和下一步练习重点`

生成后展示 4 个区块：

1. 本场表现摘要
2. 稳定优势
3. 稳定弱项
4. 下一步建议

字段映射：

1. `avg_score` -> `本场平均分`
2. `promoted_questions` -> `已回流练习题`
3. `stable_strengths` -> `你已经比较稳定的点`
4. `stable_weaknesses` -> `优先补齐的点`
5. `recommended_focus` -> `下一轮建议重点`

弱化内容：

1. `memory_path` 默认不展示给普通用户
2. `promoted_new_questions/promoted_updated_questions` 收进“查看详情”

阶段规则：

1. 进行中默认仅展示空态或简要提示
2. 全部题目完成后，应引导用户先查看该 tab
3. 复盘成功生成后，可自动打开抽屉并切到该 tab

## 7. 关键状态机

### 7.1 进行中

条件：

1. 有当前题
2. 未在提交中

页面表现：

1. 当前题卡片显示题干
2. 输入区可编辑
3. 顶部状态条显示 `等待你的回答`
4. 抽屉默认关闭，用户保持沉浸式作答视图

### 7.2 评分中

条件：

1. 用户已提交回答
2. 正在消费 SSE

页面表现：

1. 输入区禁用
2. 最近一条 AI 评价消息进入流式态
3. 顶部状态条和抽屉 `进展` tab 显示当前阶段文案

建议阶段文案映射：

1. `接收回答`
2. `检索相关资料`
3. `生成评分反馈`
4. `判断是否继续追问`

### 7.3 下一题已准备好

条件：

1. 当前轮 AI 评价已完成
2. 后端已经返回 `next_question`
3. 前端已决定不立即向聊天流插入完整下一题气泡

页面表现：

1. AI 评价完整展示并停留
2. 当前题卡片切换到下一题预备态或直接展示下一题
3. 页面主焦点从“评价气泡”转到“当前题卡片 + 输入区”
4. 聊天流只展示过渡消息，不展示完整新题大气泡

推荐交互：

1. 自动切换方案：
   - AI 评价完成后延迟 `800ms - 1200ms`
   - 当前题卡片切换为下一题
2. 半自动方案：
   - AI 评价完成后，卡片显示 `进入下一题`
   - 用户点击后切换

产品建议：

1. 默认优先采用“自动轻延迟切换”
2. 若后续用户测试发现仍然节奏过快，再升级为“点击进入下一题”

### 7.4 已插入追问

条件：

1. `next_question.question_type === follow_up`

页面表现：

1. 系统消息提示插入追问
2. 当前题卡片显示追问说明
3. 抽屉 `题目` tab 中该题高亮并带 `新插入` 标签
4. 下一题若为追问，优先通过当前题卡片切换呈现，而不是紧贴评价追加完整题目气泡

### 7.5 全部完成

条件：

1. 队列全部 answered 或 skipped
2. 无当前题

页面表现：

1. 当前题卡片切换为完成态
2. 输入区隐藏或禁用
3. 引导用户打开或自动打开 `面试面板`
4. 抽屉默认切到 `复盘` tab
5. 主 CTA 优先展示 `生成复盘`

### 7.6 复盘已生成

条件：

1. `retrospect !== null`

页面表现：

1. 抽屉 `复盘` tab 展示完整内容
2. 若会话已完成，`结束面试` 成为最后一步 CTA

## 8. 文案与标签优化

建议替换：

1. `Live Transcript` -> `面试实录`
2. `证据命中` -> `资料佐证`
3. `source llm` -> `题目来源：系统生成`
4. `medium` -> `难度：中等`
5. `生成复盘` 保留
6. `刷新队列` -> `同步题目状态`
7. `进入下一题`
8. `进入追问`
9. `面试面板`
10. `进展`
11. `题目`
12. `复盘`

辅助文案：

1. 输入区提示改为：`尽量说明背景、你的方案、关键取舍和结果指标，AI 会基于资料证据给出反馈`
2. 追问提示改为：`这道追问来自你上一轮回答中的薄弱点，建议补充证据、取舍或结果`
3. 下一题过渡提示改为：`这一题先到这里，接下来进入下一题`

## 9. 数据与接口影响

### 9.1 可直接复用的现有字段

1. `queueItems`
2. `currentQuestion`
3. `turns`
4. `latestTurn.evidence_refs_count`
5. `latestTurn.retrieval_strategy`
6. `retrospect.avg_score`
7. `retrospect.long_term_memory`

补充说明：

1. `next_question` 已经足够支持“当前题卡片切换”方案
2. 即使后端不改接口，前端也可以先停止把 `next_question` 自动追加为完整聊天气泡

### 9.2 建议新增但非强依赖字段

#### A. 会话状态字段

建议在前端本地派生或后端补充：

1. `session_stage_label`
2. `current_action_hint`

用途：

1. 给抽屉 `进展` tab 和流式评价头部使用

#### B. 追问原因字段

建议在 `turns/stream result` 中增加：

1. `follow_up_reason`

枚举示例：

1. `lack_of_evidence`
2. `missing_tradeoff`
3. `insufficient_depth`
4. `unclear_result`

用途：

1. 给“为什么插入追问”提供更强解释

#### C. 产品化检索策略字段

建议新增：

1. `retrieval_strategy_label`

示例：

1. `优先查看简历/JD`
2. `优先查看知识库资料`

用途：

1. 避免前端自行硬编码英文枚举到中文

## 10. 埋点建议

关键事件：

1. 会话页曝光
2. 当前题曝光
3. 回答提交
4. 评分完成
5. 追问插入
6. 复盘生成点击
7. 复盘生成成功
8. 结束面试点击

核心指标：

1. 单场完成率
2. 平均每题作答时长
3. 复盘生成率
4. 追问后继续作答率
5. 会话中断点分布

## 11. 交互时序

### 11.1 现状时序

当前问题时序可简化为：

1. 用户发送回答
2. AI 评价流式输出
3. `result` 返回
4. 前端立即设置 `currentQuestionId = next_question.id`
5. 聊天流立刻追加下一题气泡

问题：

1. AI 评价与下一题在视觉上挤在一起
2. 用户还没完成“上一题收口”，就被带入“下一题开始”
3. 聊天流同时承担“历史记录”和“当前任务承载”两种职责，焦点混乱

### 11.2 目标时序

推荐改为：

1. 用户发送回答
2. AI 评价流式输出
3. AI 评价完成并停留
4. 显示过渡状态：`下一题已准备好`
5. 当前题卡片切换为下一题
6. 焦点回到输入区，等待用户继续作答

### 11.3 普通下一题时序

```text
用户提交回答
-> 输入区禁用
-> AI 评价流式输出
-> AI 评价完成
-> 系统提示“这一题先到这里，下一题已准备好”
-> 延迟 800ms - 1200ms
-> 当前题卡片切换为下一题
-> 输入区恢复可编辑
-> 光标聚焦回答框
```

### 11.4 追问题时序

```text
用户提交回答
-> 输入区禁用
-> AI 评价流式输出
-> AI 评价完成
-> 系统提示“基于你刚才的回答，接下来会继续追问这个点”
-> 题目地图高亮新插入追问
-> 延迟 800ms - 1200ms
-> 当前题卡片切换为追问题
-> 输入区恢复可编辑
```

### 11.5 可选半自动时序

适用于后续如果发现用户需要更多阅读 AI 评价时间：

```text
用户提交回答
-> AI 评价完成
-> 当前题卡片显示“下一题已准备好”
-> 用户点击“进入下一题”
-> 卡片切换
-> 输入区聚焦
```

建议：

1. 第一版先做“自动轻延迟切换”
2. 若用户测试中出现“评价没看完就切题”的反馈，再升级为半自动

## 12. 前端实现建议

### 12.1 组件职责调整

当前核心组件：

1. `SessionHeader`
2. `ConversationTranscript`
3. `Composer`
4. `QueuePanel`
5. `StatusPanel`
6. `RetrospectPanel`
7. `InterviewSessionRoom`

建议新增或重构：

1. 新增 `CurrentQuestionCard`
2. 新增 `NextQuestionTransitionBanner`
3. 新增 `InterviewSideDrawer` 或 `InterviewPanelDrawer`
4. 新增 `InterviewPanelTabs`
5. 将 `ConversationTranscript` 调整为“历史消息区”
6. 将 `Composer` 与 `CurrentQuestionCard` 组合成“当前回合工作区”

### 12.2 推荐新增状态

建议在 `InterviewSessionRoom` 内增加：

1. `pendingNextQuestion: InterviewQuestion | null`
2. `transitioningToNextQuestion: boolean`
3. `stageLabel: string`
4. `questionCardMode: "active" | "transition" | "completed"`
5. `isPanelOpen: boolean`
6. `panelTab: "progress" | "queue" | "retrospect"`

含义：

1. `pendingNextQuestion`
   - 后端已返回但尚未正式进入当前回合的下一题
2. `transitioningToNextQuestion`
   - 当前正在从上一题收尾过渡到下一题
3. `stageLabel`
   - 用于展示“检索中 / 评分中 / 准备下一题”
4. `questionCardMode`
   - 控制当前题卡片是正常题目态、过渡态还是完成态
5. `isPanelOpen`
   - 控制右侧抽屉开关
6. `panelTab`
   - 控制抽屉当前所在 tab

### 12.3 推荐状态流转

提交回答后：

1. `submittingTurn = true`
2. `stageLabel = "正在评分"`
3. `questionCardMode = "active"`

收到 `result` 后：

1. 完成 AI 评价内容落定
2. 若存在 `next_question`
   - `pendingNextQuestion = data.next_question`
   - `transitioningToNextQuestion = true`
   - `questionCardMode = "transition"`
   - 不立即 `setCurrentQuestionId(data.next_question.id)`
3. 若不存在 `next_question`
   - 按完成态处理

过渡定时结束后：

1. `setCurrentQuestionId(pendingNextQuestion.id)`
2. `pendingNextQuestion = null`
3. `transitioningToNextQuestion = false`
4. `questionCardMode = "active"`
5. `stageLabel = "等待你的回答"`

全部完成后：

1. `questionCardMode = "completed"`
2. `panelTab = "retrospect"`
3. 可选：`isPanelOpen = true`

### 12.4 聊天流插入策略调整

当前策略：

1. `currentQuestion` 变化时自动往 `conversationRows` 里 append 一条 question 消息

建议调整为：

1. 历史题目仍可记录在 `conversationRows`
2. 但“下一题”不要在 AI 评价完成的同一时刻立刻 append
3. 可选两种实现：

方案 A：

1. 完全停止向聊天流追加新的完整题目气泡
2. 所有当前题只由 `CurrentQuestionCard` 承载
3. 聊天流仅保留用户回答、AI 评价、系统提示

方案 B：

1. 历史上仍保留题目消息
2. 但只在切题完成后、且与 AI 评价错开后才插入
3. 题目消息样式弱化为“题目记录”，不是主视觉焦点

产品建议：

1. 优先采用方案 A，心智更清晰
2. 如果担心历史题目缺失，再采用方案 B

### 12.5 抽屉逻辑

默认行为：

1. 桌面端以右侧 slide-over 形式出现
2. 移动端以 overlay drawer 或全屏 sheet 呈现
3. 抽屉内部固定 3 个 tab：`进展 / 题目 / 复盘`

默认 tab 建议：

1. 进行中默认打开 `题目`
2. 若系统正在评分，可保持在当前 tab，不强制跳转
3. 全部完成或复盘成功后，默认切到 `复盘`

禁止事项：

1. 不允许在抽屉中继续堆叠过多卡片形成第二个 dashboard
2. 不允许把所有低频开发信息暴露给普通用户

### 12.6 当前题卡片展示逻辑

`questionCardMode = active`：

1. 展示当前题完整题干
2. 展示题型、难度、追问标签
3. 输入区可作答

`questionCardMode = transition`：

1. 上一题评价已完成
2. 卡片展示过渡提示
3. 若 `pendingNextQuestion` 已存在，可直接展示下一题信息，但输入区暂时禁用

`questionCardMode = completed`：

1. 展示“本场题目已完成”
2. 主动作切换为“生成复盘”

### 12.7 焦点管理

切题完成后应自动：

1. 将视图滚动到当前题卡片顶部附近
2. 聚焦输入框
3. 避免用户仍停留在上一个 AI 评价气泡的位置

目的：

1. 让下一步动作非常明确
2. 减少用户重新寻找当前输入位置的成本

4. 把下一题切换主入口迁移到当前题卡片
5. 调整输入区与切题状态的禁用/启用逻辑
6. 去掉常驻右侧栏

8. 将状态、题目、复盘内容迁移进抽屉

### 13.2 P1 实现项

1. 新增 `进入下一题` 半自动模式开关或实验配置
2. 追问原因文案映射
3. 切题时的轻量动效
4. 焦点自动回到输入框

### 13.3 风险点

1. 如果完全取消聊天流里的题目消息，历史记录感可能减弱
2. 如果切题延迟过长，会让用户误以为系统卡住
3. 如果切题延迟过短，体验会退回当前问题

建议参数：

1. 默认过渡延迟：`1000ms`
2. 允许范围：`800ms - 1200ms`

## 14. 验收标准

### 14.1 体验验收

1. 用户进入页面后，3 秒内能识别当前题、当前进度、主操作。
2. 用户提交回答后，页面能持续表达“系统正在处理”，没有明显失联感。
3. 用户能看懂追问是基于上一题薄弱点插入，而不是随机新增。
4. 复盘面板在未生成前不抢主流程，生成后能给出明确行动建议。
5. AI 评价完成后，下一题不会以紧邻大气泡的形式立即打断用户阅读。
6. 用户能够明确知道“下一题在哪里看、在哪里回答”。
7. 默认页面应明显呈现为“对话作答页”，而不是“双栏控制台”。

### 14.2 前端验收

1. 不破坏现有 SSE 流式链路。
2. 不破坏现有队列刷新与自动推进逻辑。
3. 页面在 `xl` 和 `md` 尺寸下都能保持主次分明。
4. `DebugPanel` 默认不在普通用户视图中出现。
5. 切题逻辑由当前题卡片承载后，不出现重复题目展示或状态错乱。
6. 抽屉关闭时，主工作区应明显扩大并保持沉浸式布局。

### 14.3 数据验收

1. 已答题数、当前题、平均分、追问状态与后端返回保持一致。
2. 完成态、复盘态、追问态之间切换无歧义。
3. `next_question` 存在时，前端能正确经历“评价完成 -> 过渡 -> 卡片切题”流程。
## 15. 实施优先级

### P0

1. 重构页面信息层级
2. 新增当前题卡片
3. 右侧面板改为抽屉
4. 调整主要文案

### P1

1. 流式阶段状态可视化
2. 追问原因解释
3. 完成态 CTA 重构

### P2

## 16. 推荐落地顺序

1. 先做纯前端重构，不改接口契约。
2. 上线后观察完成率、复盘生成率、追问后继续作答率。
3. 再决定是否补 `follow_up_reason` 和 `retrieval_strategy_label` 等后端字段。

## 17. 结论

本次优化的核心不是“再加功能”，而是把现有能力重组成一个更像真实面试现场的产品界面：

1. 中心是当前题和作答
2. 辅助信息统一进入抽屉，不再以常驻右栏打断主流程
3. 复盘属于收尾阶段，而不是进行中常驻干扰
4. 调试信息退出默认用户视图

这样可以在不显著增加系统复杂度的前提下，提升页面的聚焦度、解释性和完成闭环能力。
