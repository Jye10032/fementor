# 模拟面试会话页实现方案

- 版本：v1.0
- 日期：2026-03-23
- 页面路径：`/interview/session`
- 对应实现：
  - `apps/web/components/interview-session-room.tsx`
  - `apps/web/app/interview/session/page.tsx`
- 对应上游文档：
  - `docs/pages/interview-session-optimization-spec.md`

## 1. 本次实现目标

本次不是继续堆功能，而是把现有会话页重组为更聚焦的面试空间。

本期目标：

1. 顶部固定展示当前题，不再依赖聊天流承载题目。
2. 聊天区只展示 AI 评价和过程输出。
3. 回答区固定在底部，保证作答路径稳定。
4. 右侧信息改为抽屉，不再常驻占宽。
5. 全部题目完成后，主界面切换为总结报告页，而不是保留聊天历史。

本期范围：

1. 仅调整前端页面结构、状态流转和文案。
2. 不修改接口契约。
3. 优先复用现有后端字段。

## 2. 已确认的产品决策

1. 顶部题目卡片始终固定可见，中间聊天区内部滚动。
2. 当前题固定在页面顶部，不作为聊天主消息展示。
3. 聊天气泡只承载 AI 评价和过程输出。
4. 评分完成后，题目卡片内出现 `进入下一题`。
5. 用户可停留阅读当前回合；若无操作，`1000ms` 后自动切题。
6. 抽屉只用于进行中的侧边查看，默认 tab 为 `当前进展`。
7. 移动端抽屉为半屏覆盖，不做全屏。
8. 全部题目完成后，页面直接切换为水平居中的总结/报告页。
9. 总结页不保留历史聊天，只保留总结内容与 `生成复盘 / 结束面试`。
10. `生成复盘` 必须手动点击触发。
11. 调试信息默认隐藏，仅开发模式可见。
12. 如果追问原因缺失且仍在生成中，前端仅显示 `正在生成`。

## 4. 目标页面结构

### 4.1 进行中态

页面采用四段结构：

1. `SessionTopBar`
2. `CurrentQuestionCard`
3. `ConversationTranscript`
4. `Composer`

右侧辅助信息改为：

1. `InterviewPanelDrawer`

布局原则：

1. 题目卡片固定在顶部。
2. 聊天区在中间独立滚动。
3. 输入区固定在底部。
4. 抽屉关闭时，主工作区尽量占满。

### 4.2 完成态

全部题目完成后，不再保留原聊天布局，改为：

1. `SessionTopBar`
2. `InterviewCompletionReport`

完成态只保留：

1. 总结信息
2. `生成复盘`
3. `结束面试`

不保留：

1. 题目卡片
2. 聊天历史
3. 输入区
4. 进行中的侧边抽屉作为主视图

## 7. 切题状态机

### 7.1 当前问题

现有实现中，在 SSE `result` 后立即：

1. 更新队列
2. `setCurrentQuestionId(data.next_question?.id || null)`
3. 由 `currentQuestion` 变化继续向聊天流追加题目

这会造成：

1. 用户尚未读完评价就被带入下一题。
2. 聊天流同时承载“上一题评价”和“下一题开始”。
3. 题目展示位置不稳定。

### 7.2 目标流转

目标流程：

1. 用户发送回答
2. 输入区禁用
3. 阶段条展示处理中状态
4. AI 评价流式输出
5. `result` 到达并固化评价
6. 若存在下一题：
   - 记录到 `pendingNextQuestion`
   - `questionCardMode = "transition"`
   - 卡片内展示 `进入下一题`
   - 启动 `1000ms` 自动切题
7. 用户点击则立即切题
8. 若用户无操作，则定时自动切题
9. 切题完成后恢复输入区并聚焦输入框

### 7.3 过渡态规则

`questionCardMode = "transition"` 时：

1. 题目卡片不再是纯展示态。
2. 卡片显示过渡提示和 `进入下一题`。
3. 输入区保持禁用。
4. 用户可以继续阅读当前回合评价，但不能在该阶段继续提交输入。

### 7.4 完成态规则

若 `result` 后不存在 `next_question`，且队列已全部完成：

1. `questionCardMode = "completed"`
2. `showCompletionReport = true`
3. 退出聊天工作区

## 8. SSE 与阶段文案

### 8.1 展示原则

阶段文案不进入聊天气泡，统一展示在顶部阶段条。

### 8.2 映射建议

前端可先做以下映射：

1. `idle` -> `等待你的回答`
2. `receiving` -> `正在接收回答`
3. `retrieving` -> `正在检索相关资料`
4. `evaluating` -> `正在生成评分反馈`
5. `deciding` -> `正在判断是否继续追问`
6. `generating_followup` -> `正在生成`
7. `transition` -> `下一题已准备好`

### 8.3 SSE 处理调整

当前 `stage` 事件会在 token 流开始前占用评价气泡内容。

建议调整为：

1. `stage` 事件优先更新 `stageLabel`
2. `token` 事件持续写入评价气泡
3. `result` 事件固化评价内容并启动切题状态机

## 9. 聊天流调整

### 9.1 保留内容

1. 用户回答
2. AI 评价
3. 系统提示

### 9.2 移除内容

1. 当前题自动插入聊天流
2. 下一题自动以完整大气泡出现

### 9.3 代码影响

当前这段逻辑需要删除或重写：

1. 监听 `currentQuestion` 自动 append `question` 消息
2. `askedQuestionIdsRef` 对题目消息的去重机制

## 10. 抽屉改造方案

### 10.1 默认行为

1. 抽屉默认关闭
2. 用户点击 `面试面板` 打开
3. 默认 tab 为 `当前进展`
4. 移动端为半屏 overlay

### 10.2 `当前进展` tab

迁移现有 `StatusPanel` 内容，优先使用现有字段：

1. 已完成题目数
2. 当前会话平均分
3. 最近一轮资料佐证数
4. 当前状态文案
5. 当前状态提示

字段映射建议：

1. `证据命中` -> `资料佐证`

### 10.3 `题目` tab

迁移现有 `QueuePanel`，但改成更紧凑的题目地图。

规则：

1. 当前题高亮
2. 追问单独标识
3. 新插入题目高亮
4. 状态全部改为中文

### 10.4 `复盘` tab

迁移现有 `RetrospectPanel`。

规则：

1. 未生成时显示空态说明
2. 已生成时展示现有 `retrospect` 字段
3. 不默认在进行中抢主焦点

### 10.5 调试信息

`DebugPanel` 不再进入普通用户视图。

建议：

1. 仅在开发模式下可见
2. 放入抽屉内的开发专用区域，或保持隐藏实现

## 11. 完成态报告页方案

### 11.1 页面职责

在所有题目完成后，报告页替代聊天界面，承担收尾动作。

### 11.2 展示内容

第一版仅迁移现有后端字段与现有右栏关键信息：

1. 已完成题数
2. 平均分
3. 最近一轮资料佐证或已有统计字段
4. 当前状态总结
5. 若已有 `retrospect`，展示复盘摘要

### 11.3 主动作

1. `生成复盘`
2. `结束面试`

规则：

1. 若未生成复盘，主按钮优先为 `生成复盘`
2. 生成复盘成功后，在同一报告页内刷新展示
3. 不提供返回历史聊天的主入口

## 14. 第一版之后的后续步骤

### 14.1 第一版上线后的观察项

第一版上线后，先不急着继续加交互，先验证以下问题：

1. 用户是否能在进入页面后快速理解“当前题在哪里、回答入口在哪里”。
2. `1000ms` 自动切题是否过快，是否出现“评价还没读完就切题”的反馈。
3. 题目退出聊天流后，用户是否会出现“历史题目不方便回看”的反馈。
4. 抽屉默认 `当前进展` 是否满足主要查看需求，还是更常被切到 `题目`。
5. 完成态总结页是否足够支撑用户继续点击 `生成复盘`。

建议上线后重点看这些指标：

1. 单场完成率
2. 平均每题停留时长
3. 评分完成后的继续作答率
4. `生成复盘` 点击率
5. 会话中断点分布

### 14.2 第二版

第二版重点不是继续大改布局，而是补足“状态解释”和“过渡质量”。

建议内容：

1. 把 `stage` 展示从单行文案升级为更清晰的阶段进度表现。
2. 增强“下一题已准备好”的过渡反馈，让用户更清楚当前处于“阅读完成 -> 即将切题”。
3. 若验证发现 `1000ms` 自动切题偏快，则把混合模式升级为更偏手动：
   - 默认停留
   - 明确按钮进入下一题
   - 自动切题仅作为兜底或实验开关
4. 补切题后的滚动与焦点管理，确保视图自动回到题目卡片和输入区。
5. 完善移动端半屏抽屉的手势和关闭体验。

第二版可新增的前端状态：

1. `autoAdvanceEnabled`
2. `autoAdvanceDelay`
3. `stageProgressItems`
4. `hasUserReviewedEvaluation`

### 14.3 第三版

第三版重点处理“解释性字段产品化”和“复盘收尾质量”。

建议内容：

1. 接入后端补充字段：
   - `follow_up_reason`
   - `session_stage_label`
2. 将追问解释从通用状态升级为明确原因说明。
3. 让完成态总结页与复盘结果更自然衔接，减少“先完成，再点复盘，再看结果”的割裂感。
4. 把现有右侧抽屉中的 `复盘` 内容与完成态总结页做信息对齐，避免两个区域表达冲突。
5. 对完成态的指标卡做进一步产品化包装，但仍基于现有后端字段，不急于新增数据结构。

第三版的目标是：

1. 用户不仅知道系统在做什么，还知道“为什么这样做”。
2. 追问、评分、资料佐证这些能力从技术字段变成自然语言反馈。
3. 复盘从“结果展示”升级为“下一步行动建议入口”。

### 14.4 第四版及以后

第四版以后再考虑扩展交互能力，而不是在第一轮重构中一次塞满。

可进入后续排期的方向：

1. `要求澄清题意`
2. `跳过此题`
3. `追问面试官`
4. 题目详情展开能力
5. 更细的报告页分区与行动建议布局
6. 更细的实验开关：
   - 自动切题开关
   - 题目地图默认 tab
   - 完成态报告样式实验

这些能力建议放后面做的原因：

1. 当前核心问题仍是页面聚焦和状态解释，不是缺按钮。
2. 如果第一版的信息架构还没稳定，继续加操作只会重新变回控制台。
3. 等主流程稳定后，再判断哪些二级能力真的值得产品化。

3. 优化移动端抽屉交互

交付判断标准：

2. 提升追问解释度

## 16. 风险点

1. 题目不再进入聊天流后，历史题目回看能力会减弱。
2. `loadQuestionQueue()` 与本地过渡态可能互相覆盖，需要避免状态竞争。
3. 完成态判断必须考虑 `pendingNextQuestion`，否则可能提前进入总结页。
4. `Composer` 禁用条件需要同步扩展到 `transition` 和完成态。

## 17. 验收清单

### 15.1 页面体验

1. 进入页面后，用户能立即看到当前题、当前进度和主操作。
2. 聊天区滚动时，题目卡片仍固定可见。
3. 评分完成后，用户能在卡片中看到 `进入下一题`。
4. 用户不点时，`1000ms` 后自动切题。
5. 切题后焦点回到输入区。
6. 抽屉关闭时，主工作区明显变宽。
7. 全部完成后，聊天界面被总结页替代。

### 15.2 数据一致性

1. `currentQuestion` 与队列状态一致。
2. `next_question` 存在时，前端正确经历“评价完成 -> 过渡 -> 切题”。
3. `retrospect` 生成后可直接在报告页或抽屉中展示。

### 15.3 开发约束

1. 不改接口契约。
2. 不破坏现有 SSE 链路。
3. 调试信息默认不出现在普通用户视图。
