# 用户笔记检索与 Rubric 对照 Spec

> **当前状态：尚未实施。** 代码中未出现 `rubric_refs`、`note_refs`、`note_match_status`、`conflict_flags`、`suggested_note` 等结构，评分链路与用户知识比对链路尚未分离。属于远期规划，依赖面试单轮链路整合先完成。

## 背景

当前系统同时具备以下能力基础：

- 用户可上传简历、JD、学习笔记、项目文档
- 系统可生成模拟面试题与练习题
- 检索链路已接入 Sirchmunk
- 评分链路已具备 LLM 生成反馈能力

在学习型场景下，技术八股或知识题的目标并不只是“像真实面试官一样打分”，还应包括：

- 判断用户是否真正掌握自己学过的内容
- 判断当前回答与已有笔记是否一致
- 判断用户笔记中的记录是否与标准知识点存在冲突
- 在知识库未覆盖时，自动生成一份建议笔记，帮助形成学习闭环

因此，Sirchmunk 在该场景中的职责，不应是直接定义评分标准，而应作为“用户知识证据检索层”融入评分与反馈链路。

## 目标

- 使用 Sirchmunk 检索用户笔记中是否已有与当前题目相关的记录
- 使用系统内置 rubric 作为稳定评分标准
- 在无笔记命中时，基于 rubric 为用户生成一份候选建议笔记
- 在有笔记命中时，判断：
  - 当前回答是否覆盖已有笔记中的关键点
  - 当前回答是否与已有笔记冲突
  - 用户笔记是否与 rubric 冲突
- 将“建议补充”与“冲突提示”变成可视化学习反馈，而不是单纯打分

## 非目标

- 不让 Sirchmunk 定义评分标准
- 不让 LLM 直接覆盖用户原始笔记
- 不将 rubric 和用户笔记混在同一类 evidence 中使用
- 不将完整长回答原文直接作为 Sirchmunk 主查询输入

## 架构边界

- 面试评分的标准来源固定为 `rubric + expected_points + LLM reasoning`。
- 用户上传的笔记、项目文档、知识库不作为评分标准来源。
- Sirchmunk 仅用于检索用户已有知识记录，服务于知识覆盖判断、冲突提示与候选建议笔记生成。
- 评分依据与用户知识依据必须分开存储、分开返回、分开展示。
- LLM 生成的标准化建议内容默认只进入候选区，不直接覆盖用户正式知识库。

## 证据分层规则

建议将证据明确拆成两类：

- `rubric_refs`
  - 用于评分标准、expected points 命中、扣分说明
- `note_refs`
  - 用于用户笔记命中、历史记录比对、冲突提示

约束：

- 评分结果必须优先引用 `rubric_refs`，不能因为 `note_refs` 命中而默认判定“回答正确”。
- `note_refs` 只说明“用户曾记录过什么”，不说明“该记录一定正确”。
- 如果 `rubric_refs` 缺失，不允许输出强事实判断，只能输出“标准依据不足”。
- 如果 `note_refs` 缺失，可以继续评分，但需要补充“用户知识库未覆盖该知识点”的学习反馈。

## 核心角色划分

## 1. Rubric 层

负责回答：

- 这题在标准上应该怎么评
- 关键知识点是什么
- 哪些信号代表回答较强
- 哪些信号代表回答存在问题

来源建议：

- 系统内置题型 rubric
- 题目级 expected points
- JD / competency model

## 2. 用户笔记检索层

由 Sirchmunk 负责，回答：

- 用户自己的知识库里有没有与当前问题相关的记录
- 命中的笔记片段是什么
- 笔记记录的关键点是什么

来源包括：

- 学习笔记
- 技术八股记录
- 项目总结
- 上传的知识文档

## 3. 推理与反馈层

由 LLM 负责，结合：

- 当前题目
- 用户回答
- rubric
- 检索到的用户笔记 evidence

输出：

- 得分与反馈
- 覆盖不足点
- 冲突提示
- 候选建议笔记

## 核心链路

### 场景 A：用户笔记未命中

1. 用户回答当前问题
2. 系统读取题目对应 rubric / expected points
3. Sirchmunk 检索用户知识库
4. 未检索到相关笔记
5. LLM 基于 rubric 与当前回答生成：
   - 当前回答评价
   - 建议回答总结
   - 一条候选建议笔记
6. 系统将候选建议笔记写入候选记忆区，而非直接覆盖原始笔记

### 场景 B：用户笔记已命中

1. 用户回答当前问题
2. 系统读取题目对应 rubric / expected points
3. Sirchmunk 检索出相关用户笔记
4. LLM 对比三者：
   - 用户当前回答
   - 用户已有笔记
   - rubric 标准点
5. 输出：
   - 当前回答是否覆盖笔记中的关键点
   - 当前回答与笔记是否存在冲突
   - 用户笔记与 rubric 是否存在冲突
6. 若存在冲突，提示用户复核，而不是直接改写原笔记

## 关键判断逻辑

## 一、回答与 Rubric 的关系

判断：

- 是否命中标准关键点
- 哪些 expected points 已覆盖
- 哪些关键点缺失
- 当前回答是否存在明显概念错误

## 二、回答与用户笔记的关系

判断：

- 当前回答是否覆盖到已有笔记中的关键内容
- 当前回答是否与用户历史记录不一致

适合输出：

- “你当前回答没有覆盖到笔记中的关键点”
- “当前回答与已有笔记中的表述存在差异”

## 三、用户笔记与 Rubric 的关系

判断：

- 用户笔记中的表述是否与标准点冲突
- 用户笔记是否只覆盖部分知识点
- 用户笔记是否存在概念混淆、过时或偏差

适合输出：

- “你的知识笔记中关于该知识点的记录可能与当前标准点不完全一致，建议重新核对”
- “笔记中的表述与本题考察要点存在冲突，建议回看并修订”

## 候选笔记写回策略

## 不直接覆盖原始笔记

LLM 生成的建议笔记默认只写入候选区，不直接改原文。

原因：

- 标准答案不一定完全适合用户当前的笔记组织方式
- LLM 生成内容存在误差风险
- 冲突提示应先由用户确认

## 建议笔记的来源约束

- `suggested_note` 的内容来源应标记为 `rubric_derived` 或 `llm_refined_from_rubric`。
- 不允许把用户当前回答原样当作标准建议笔记写回。
- 不允许把 Sirchmunk 命中的旧笔记片段直接重写为“标准答案”。
- 当建议笔记来自 LLM 生成时，应同时保留触发原因，例如：
  - 当前知识库未命中
  - 当前笔记与 rubric 冲突
  - 当前回答覆盖不足

## 建议写入位置

候选内容可写入：

- `memory`
- `review_notes`
- `pending_note_updates`

示例结构：

```json
{
  "topic": "rbac_authorization",
  "status": "suggested",
  "source_question": "在 NewsSystemPro 中，你是怎么实现 RBAC 权限控制的？",
  "suggested_note": "RBAC 的核心是将权限先赋给角色，再将角色赋给用户，并通过统一鉴权链路控制接口访问。",
  "reason": "当前知识库未命中相关记录"
}
```

## Sirchmunk 融入方式

## Sirchmunk 负责查什么

Sirchmunk 只查用户资料，不负责定义标准。

适合检索的对象：

- 用户知识笔记
- 用户项目总结
- 用户学习记录
- 用户上传的技术文档

## Sirchmunk 不负责什么

- 不负责定义 rubric
- 不负责决定 expected points
- 不负责直接给分

## 推荐输入

Sirchmunk 主查询应使用：

- 题目级 `retrieval_query`
- 题目级 `keywords`
- 回答的轻量摘要

不建议使用：

- 面试题原文 + 完整长回答原文

因为这容易使 Sirchmunk 将输入判为 `DocQA / explain`。

## 适用场景

### 学习模式

最适合启用本 spec：

- 技术八股题
- 基础概念题
- 用户学习笔记驱动的问题

目标是：

- 检验用户是否掌握自己整理过的知识
- 帮助用户发现知识冲突
- 自动补全未覆盖的笔记

### 面试模拟模式

该 spec 可作为辅助能力存在，但不应压过通用面试评价。

项目题或开放题中：

- rubric 仍为主标准
- 用户笔记检索仅作为辅助证据

## 风险与注意事项

- Rubric 必须足够稳定，否则冲突判断没有依据
- 用户笔记命中不足时，不能轻易断言“笔记错误”
- 冲突提示应使用保守表达，避免绝对化结论
- 建议笔记默认进入候选区，不能直接覆盖原笔记
- 需要区分：
  - `answer_vs_rubric`
  - `answer_vs_notes`
  - `notes_vs_rubric`

## 实施顺序建议

1. 为知识题建立最小 rubric / expected points 结构
2. 为题目增加 `retrieval_query / keywords / topic`
3. 在回答后新增 Sirchmunk 用户笔记检索
4. 新增结构化输出：
   `note_match_status / conflict_flags / suggested_note`
5. 前端增加：
   - 笔记未命中提示
   - 笔记冲突提示
   - 建议笔记保存入口
6. 后续再扩展到：
   - 候选笔记确认
   - 笔记修订流程
   - 多轮知识闭环追踪
