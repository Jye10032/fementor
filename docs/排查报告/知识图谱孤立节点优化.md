# 知识图谱孤立节点优化

日期：2026-04-02

## 问题

线上 Supabase 14 篇面经生成了 266 个去重知识点节点，大量孤立节点无法形成有效图谱关联。

根因：
1. 骨架 skeleton.json 只有 10 个一级分类（全是技术类），缺少「行为面」「项目」「安全」「网络」等高频分类
2. LLM 每篇面经独立标注，自由造词导致同义碎片化（如「防抖与节流」vs「防抖」vs「防抖作用」）
3. annotation prompt 无词表约束，LLM 每次措辞不一致

## 改动

### 1. 扩充骨架 `apps/api/data/knowledge-graph-skeleton.json`

新增 6 个一级分类：网络、安全、数据库、行为面、项目、AI。补充现有分类子节点（Vue 加 Vue3/组件通信，浏览器加 IndexedDB/虚拟列表等）。

- 一级分类：10 → 16
- 骨架覆盖节点：175 → 215+

### 2. 归一化层 `apps/api/src/experience/knowledge-graph.js`

两层归一化机制：

- **文本标准化 + 骨架词表精确匹配**：去空格/连字符，大小写不敏感匹配骨架词表（`虚拟 DOM`→`虚拟DOM`，`Tree-Shaking`→`Tree Shaking`）
- **fuzzyMatchSkeleton 子串模糊匹配**：知识点包含骨架节点名时自动归并（长词优先匹配避免误伤，如 `CSRF攻击`→`CSRF`）
- **inferOrphanParents 增强**：先查骨架 children 反向索引 → 再查文本标准化匹配 → 最后用 related 权重推断

早期方案使用了 SYNONYMS 手动映射表（~120 条），后因无法区分同义词和父子关系（如 `双 Token` 不等于 `JWT`，是其子概念）且难以维护，已删除。改为纯文本标准化 + 模糊匹配，语义层面交给 LLM 源头约束和骨架层级处理。

应用位置：
- `buildCooccurrenceFromStore`：图谱全量构建时归一化
- `updateGraphIncremental`：增量更新时归一化
- `assemblePipelineResult`（cleaner.js）：写入 DB 前归一化

### 3. 源头约束 `apps/api/src/experience/cleaner.js`

annotation prompt 注入骨架词表（212 词 / ~1500 字符），指令从"优先使用"改为"必须从列表中选取，仅当确实没有匹配项时才可新建"。

## 效果

对 14 篇面经执行 reclean 后：

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 去重知识点 | 266 | 197 |
| 高频节点聚合 | 性能优化×8, AI×2 | 性能优化×22, AI×13 |
| 骨架覆盖率 | ~30% | ~65% |

## 后续可优化

- 批量 annotation：一批面经合并送入 LLM，同一上下文内标签自然复用
- 自动骨架扩充：共现频次超阈值的孤立节点自动挂到最相关一级分类
- 动态 parent 调整：数据量上来后根据共现权重变化重新推断节点归属

## 部署问题修复

### skeleton.json 未被 git 追踪

线上报错：`[knowledge-graph.skeleton.failed] ENOENT: no such file or directory, open '/app/data/knowledge-graph-skeleton.json'`

根因：`.gitignore` 中 `data/` 规则忽略了所有 data 目录，`apps/api/data/knowledge-graph-skeleton.json` 未提交到仓库，线上部署时文件不存在。

修复：在 `.gitignore` 中为 `apps/api/data/` 添加例外，允许 `knowledge-graph-skeleton.json` 被追踪。

### 骨架子节点未展开为图谱节点

部署后图谱只有 33 个节点（骨架 JSON 的顶层 key），而非预期的 215 个。

根因：`mergeSkeletonWithCooccurrence` 只为骨架 JSON 中有独立定义（有自己的 `{ parent, children, related }` 对象）的节点创建 graph entry。大量子节点（如 XSS、JWT、Vue3）只出现在父节点的 `children` 数组中，没有独立定义，必须依赖共现数据从 DB 加载后才会被创建。当共现数据未加载（API 刚启动、DB 连接慢等），这些子节点就不存在。

初始修复：在 `mergeSkeletonWithCooccurrence` 中遍历骨架所有 children，主动创建 graph entry。但这导致 215 个预定义节点全部出现在图谱中，包括大量从未在面经中出现过的概念（如「柯里化」「IIFE」「双缓冲」「贪心」）。

最终修复：重构 `mergeSkeletonWithCooccurrence` 的节点创建逻辑，改为数据驱动：

- 骨架只提供结构定义（parent/children 层级关系）和词表约束，不自动生成图谱节点
- 只有面经数据中实际出现的标签才会创建图谱节点
- 节点创建后，从骨架查询其 parent/children/related 信息来建立层级关系
- 如果子节点存在但其父分类节点不在共现数据中，父节点会被补充创建（source 标记为 `skeleton`）

每个节点增加 `source` 字段标记来源：
- `both`：骨架有定义 + 面经中出现（有结构 + 有数据）
- `cooccurrence`：骨架中没有，纯从面经提取的新概念
- `skeleton`：自身无面经数据，但因子节点存在而被补充创建的父分类

14 篇面经数据下的实际效果：208 个节点，其中 157 个 both、51 个 cooccurrence、0 个纯 skeleton。

### web 端新增分类配色

`apps/web/app/graph/page.tsx` 的 `PINNED_COLORS` 补充了新增 6 个一级分类（TypeScript、网络、安全、数据库、行为面、项目、AI）的固定配色，`EXTRA_PALETTE` 备用色替换为与固定色不重复的颜色。
