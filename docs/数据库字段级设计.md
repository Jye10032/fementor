# 数据库字段级设计（MVP）

- 版本：v1.3
- 日期：2026-03-27

## 说明

当前项目处于“本地 SQLite 持续可用 + Railway 线上逐步接入 PostgreSQL”的过渡阶段。

本文档分两部分：

1. 本地现有表：当前已在 SQLite 中使用。
2. 线上新增表：用于 `Clerk + PostgreSQL + OCR 配额/缓存` 接入。

短期策略：

1. 本地开发可继续保留 `user_profile` 等表。
2. 线上正式用户体系引入 `users` 作为业务用户主表。
3. 后续业务表逐步从 `user_profile.id` 迁移到 `users.id`。

## interview_question

用途：一场模拟面试的题目队列。会话开始时一次生成，按 `order_no` 顺序提问。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| session_id | text | 关联 interview_session |
| order_no | integer | 队列顺序，从 1 开始 |
| source | text | 题目来源：`resume/doc/llm` |
| question_type | text | 题型：`basic/project/scenario/follow_up` |
| difficulty | text | 难度：`easy/medium/hard` |
| stem | text | 题干 |
| expected_points_json | text | 预期考点数组 |
| resume_anchor | text | 简历命中片段摘要，便于解释为何出这题 |
| source_ref | text | 来源标记：`resume_summary/job_description/llm_generated` |
| status | text | `pending/asked/answered/skipped` |
| created_at | text | 创建时间 |
| updated_at | text | 更新时间 |

### 设计规则

1. `source` 只表示出题来源，不表示评分证据来源。
2. `order_no` 决定默认出题顺序，不允许同一 session 重复。
3. `status=asked` 表示当前题；`answered` 表示已完成；`pending` 表示队列中尚未开始。
4. `expected_points_json` 用于后续追问、评分解释、题单沉淀，不要求前端直接展示。
5. `resume_anchor` 允许为空；当题目来自简历定制时应尽量填写。
6. `source_ref=job_description` 表示题目主要由当前岗位 JD 驱动。

### 追问插入规则

1. 当 `interview_turn.score < 70` 或 `weaknesses_json` 非空时，可触发动态追问。
2. 追问会插入到当前题后的 `order_no + 1`，并写入 `interview_question.question_type = follow_up`。
3. 同一道主问题最多插入一条追问，避免队列无限膨胀。
4. `question_bank` 回流时必须保留 `source_question_id/source_question_type/source_question_source`，这样章节练习才能知道题目来自哪一类面试问题。

### 迁移说明

`question_bank` 当前仍是有效表，但长期目标是把它拆分为：

1. `question_source`
2. `user_question_bank`
3. `question_attempt`

在完成读路径切换之前，`question_bank` 仍可作为兼容层保留。

## question_source

用途：公共题源层，统一承接面经题、模拟面试题、简历/JD 抽取题和手工题。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text/uuid | 主键 |
| source_type | text | `experience/interview/resume/jd/manual/practice` |
| source_ref_id | text | 来源对象 ID，例如 `experience_question_item.id` |
| canonical_question | text | 规范题干 |
| question_text | text | 展示题干 |
| normalized_question | text | 标准化文本，供检索和归并使用 |
| category | text | 题目分类 |
| difficulty | text | easy/medium/hard |
| track | text | 赛道，例如 `frontend` |
| chapter | text | 章节，例如 `javascript/react/project` |
| knowledge_points_json | text/jsonb | 知识点数组 |
| expected_points_json | text/jsonb | 预期考点数组 |
| metadata_json | text/jsonb | 来源附加信息 |
| status | text | `active/merged/archived` |
| merged_into_source_id | text nullable | 被合并后的目标 source id |
| created_at | text/timestamptz | 创建时间 |
| updated_at | text/timestamptz | 更新时间 |

### 设计规则

1. `source_type + source_ref_id` 应唯一，保证能回溯原始来源。
2. `canonical_question` 是公共题归一化锚点，不存用户掌握状态。
3. 被归并的题源应通过 `status=merged + merged_into_source_id` 表达，不直接删除。

## resume_parse_usage

用途：记录 OCR 配额流水，用于实现“PDF/图片每天 1 次”的限额策略。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid/text | 主键 |
| user_id | uuid/text | 关联 `users.id` |
| file_hash | text | 文件内容 hash |
| source_type | text | `pdf/image/docx/text` |
| engine | text | `volcengine/local/manual` |
| status | text | `success/failed/cached/blocked` |
| charged | boolean/int | 是否消耗 OCR 配额 |
| failure_reason | text | 失败或阻断原因，可空 |
| created_at | timestamptz/text | 创建时间 |

### 设计规则

1. 只有 `source_type in (pdf, image)` 才参与高成本 OCR 配额判断。
2. `charged=true` 表示本次真正消耗了 OCR 配额。
3. `status=cached` 时通常 `charged=false`。
4. `status=failed` 且为平台错误时，建议 `charged=false`。
5. 查询“当天是否还能 OCR”时，以 `user_id + created_at(自然日) + charged=true` 为准。

## experience_sync_job

用途：记录一次“按关键词同步面经”的任务状态，承载前端同步进度展示与后端异步任务追踪。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| user_id | text | 触发任务的用户 ID |
| keyword | text | 搜索关键词 |
| status | text | `pending/running/completed/failed` |
| requested_limit | integer | 本次请求上限，MVP 默认 10 |
| created_count | integer | 本次成功新增面经数量 |
| skipped_count | integer | 本次跳过数量（重复 / 非近 7 日 / 低质量） |
| failed_count | integer | 本次失败数量 |
| started_at | text | 任务开始时间 |
| finished_at | text | 任务结束时间 |
| error_message | text | 任务级错误信息，可空 |
| created_at | text | 创建时间 |
| updated_at | text | 更新时间 |

### 设计规则

1. `status=pending` 表示任务已创建但尚未执行。
2. `status=running` 表示当前正在抓取或清洗。
3. `status=completed` 表示任务正常结束，即使存在部分单条失败。
4. `status=failed` 表示任务级失败，通常是抓取入口、数据库或 LLM 服务整体异常。
5. 前端状态轮询只读该表，不直接感知内部抓取细节。

## experience_post

用途：记录一篇原始面经及其帖子级结构化结果，是面经库列表与详情页的主表。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| source_platform | text | 来源平台，当前固定为 `nowcoder` |
| source_post_id | text | 平台稳定帖子 ID |
| source_url | text | 原文 URL |
| keyword | text | 触发同步时的关键词 |
| title | text | 标题 |
| author_name | text | 作者名称，可空 |
| published_at | text | 原帖发布时间 |
| content_raw | text | 原始正文 |
| content_cleaned | text | LLM 清洗后的正文 |
| summary | text | 清洗摘要 |
| company_name | text | 推断出的公司名，可空 |
| role_name | text | 推断出的岗位名，可空 |
| interview_stage | text | `一面/二面/HR面/实习/校招/社招/未知` |
| quality_score | integer | 0-100，可作为展示和筛选参考 |
| is_valid | integer | 0/1，表示是否可用于前台和检索 |
| clean_status | text | `pending/completed/failed` |
| crawl_job_id | text | 关联 `experience_sync_job.id` |
| content_hash | text | 原始正文 hash，用于辅助去重 |
| created_at | text | 创建时间 |
| updated_at | text | 更新时间 |

### 索引与唯一键

1. 唯一键：`source_platform + source_post_id`
2. 索引：`published_at`
3. 索引：`company_name + role_name + published_at`
4. 索引：`crawl_job_id`

### 设计规则

1. 一条 `experience_post` 必须保留原始正文，清洗结果不能覆盖原文。
2. `is_valid=0` 的数据允许落库，但默认不进入前台主列表。
3. `clean_status=failed` 时允许仅存原始帖子，后续可重试清洗。
4. 去重优先依赖 `source_post_id`，`content_hash` 仅作辅助。

## experience_question_group

用途：表示一篇面经里的一组相关问题，用于表达主题簇、主问题与追问链的聚合关系。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| post_id | text | 关联 `experience_post.id` |
| topic_cluster | text | 主题簇，如“JavaScript 异步编程模型” |
| canonical_question | text | 该组的规范主问题 |
| group_order | integer | 在整篇面经中的顺序 |
| group_type | text | `single/chain/mixed` |
| frequency_score | real | 高频统计分，MVP 可先默认 0 |
| confidence | real | LLM 对该题组结构判断的置信度 |
| created_at | text | 创建时间 |
| updated_at | text | 更新时间 |

### 设计规则

1. 一篇面经可关联 0 到多个问题组。
2. `group_type=single` 表示只有单题，不形成明显追问链。
3. `group_type=chain` 表示存在清晰主问题和 follow-up / probe。
4. `confidence` 低时允许降级为多个平级问题项，不强行构造树。

## experience_question_item

用途：问题组中的单个问题项，是检索召回、模拟面试联动和后续高频统计的核心粒度。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| group_id | text | 关联 `experience_question_group.id` |
| post_id | text | 冗余关联 `experience_post.id`，便于查询 |
| question_text_raw | text | 原始问题表述 |
| question_text_normalized | text | 规范化问题表述 |
| question_role | text | `main/follow_up/probe/compare/scenario` |
| order_in_group | integer | 在题组中的顺序 |
| parent_item_id | text | 指向父问题项，可空 |
| category | text | `JavaScript/React/Vue/CSS/浏览器/工程化/项目/算法/行为面/其他` |
| difficulty | text | `easy/medium/hard` |
| follow_up_intent | text | `clarify/deepen/compare/verify/scenario` |
| expected_points_json | text | 预期考点数组 |
| knowledge_points_json | text | 知识点数组 |
| embedding_id | text | 预留 embedding 记录引用，可空 |
| created_at | text | 创建时间 |
| updated_at | text | 更新时间 |

### 索引建议

1. 索引：`post_id`
2. 索引：`group_id + order_in_group`
3. 索引：`question_role`
4. 索引：`category + difficulty`

### 设计规则

1. `question_role=main` 的问题项 `parent_item_id` 必须为空。
2. `follow_up/probe/compare` 应尽量指向明确的 `parent_item_id`。
3. 如果 LLM 无法稳定判断父子关系，允许 `parent_item_id` 为空，但仍保留 `group_id`。
4. 检索与模拟面试联动优先基于 `question_text_normalized`，不是 `question_text_raw`。

## experience_knowledge_point

用途：记录问题项对应的知识点，用于细粒度检索、高频统计与评分覆盖分析。MVP 可先不拆表，后续再从 JSON 中拆出。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| question_item_id | text | 关联 `experience_question_item.id` |
| knowledge_point | text | 知识点名称 |
| weight | real | 该知识点在该问题中的权重 |
| created_at | text | 创建时间 |

### 设计规则

1. MVP 若不拆表，可先保留在 `knowledge_points_json`。
2. 当需要问题级统计或评分覆盖分析时，再拆出独立表。
| summary | text | 简历摘要 |
| parse_meta | jsonb/text | 解析路径、OCR 信息、fallback 信息 |
| original_filename | text | 首次上传文件名 |
| created_at | timestamptz/text | 创建时间 |
| updated_at | timestamptz/text | 更新时间 |

### 设计规则

1. `file_hash` 必须唯一。
2. 只要 `file_hash` 命中，就可以直接复用 `parsed_text/summary/parse_meta`。
3. `parse_meta` 需要至少保留：
   - `parser`
   - `used_ocr`
   - `quality`
   - `original_filename`
4. 命中缓存不应重复扣 OCR 配额。
