# MVP 方案记录

## 2026-03-30 v0.27

### 本次调整

1. 新增《模拟面试每日额度与会话级 LLM Key 方案 Spec》，明确线上模拟面试改为“每用户每天 1 次免费”。
2. 新增《模拟面试每日额度与会话级 LLM Key Todo List》，把免费额度、错误码、会话级 Key、前端配置面板拆成可执行阶段。
3. 明确当日免费次数用完后，用户若要继续发起模拟面试，必须配置自己的 LLM Key。
4. 明确用户自带 Key 采用“后端会话内存态”：不落数据库、不回填前端、不写浏览器持久化存储。

### 这样做的原因

1. 平台默认 LLM 成本需要有清晰的产品化边界，不能无限兜底。
2. 用户可以继续使用自己的 Key，但不应该继续沿用浏览器持久化明文的方案。
3. 会话级内存态比长期落库更符合当前阶段“不长期代管用户密钥”的责任边界。

## 2026-03-28 v0.26

### 本次调整

1. 新增《公共题源运行模式与权限方案》，明确 `cloud/local` 两套运行模式。
2. 明确线上只有登录 `admin` 账号才可执行题源拉取与同步，且写入远程数据库。
3. 明确线上普通用户与未登录用户只读远程公共题源，不可触发写操作。
4. 明确线下自部署默认本机为 `admin`，拉取后只写本地 SQLite，不写远程数据库。

### 这样做的原因

1. 公共题源能力需要区分“谁能写、写到哪里、谁只能读”，否则线上和线下会混成一套模糊逻辑。
2. 线上普通用户不应具备抓取牛客或写远程数据库的能力。
3. 线下部署的核心价值是本地自治和本地可写，不应依赖线上远程数据库。

2. 在《数据库字段级设计》中补充 `question_source`、`user_question_bank`、`question_attempt` 的字段与规则。

2. 题库要同时支撑本地 SQLite 和线上 PostgreSQL，必须先把 repository 边界和迁移路径写清楚。

## 2026-03-27 v0.24

### 本次调整

1. 新增《面经库与模拟面试联动方案》，明确 `面经库` 作为新模块接入当前产品闭环。
2. 新增《面经库开发任务清单》，把方案拆成分期、表结构、接口、前端页面和联动任务。
3. 新增《题库三层模型方案》，明确题库后续应拆分为 `question_source -> user_question_bank -> question_attempt` 三层。
2. 明确“牛客抓取 -> 去重 -> LLM 结构化清洗 -> 入库 -> 检索召回 -> 模拟面试联动”的完整链路。
3. 明确数据模型采用 `experience_post -> experience_question_group -> experience_question_item` 三层结构，不使用无限嵌套 JSON 树。
4. 明确追问关系的主判断由 LLM 负责，采用“规则切片 + LLM 判关系 + 规则校验”的方案。
5. 明确检索方向不采用 `LIKE` demo 方案，而是朝“结构化过滤 + 全文检索 + 向量检索 + 高频补强 + 重排”的混合召回路线演进。

### 这样做的原因

1. 面经功能如果只停留在抓帖和列表展示，无法自然融入当前“模拟面试 -> 复盘 -> 练习”的产品主线。
2. 面经真正有价值的部分是“近期真实问题、问题簇和追问链”，而不是原始帖子全文。
3. 先把存储粒度、追问关系和检索路径设计清楚，后续实现时才能避免做成 demo 级搜索和孤立页面。
4. 当前 `question_bank` 同时承担题目定义、用户状态和来源追溯三种职责，后续必须拆层，否则面经题、模拟面试题和文档题很难统一汇流。

3. 先把迁移边界和阶段写清楚，后续实现时更容易控制回归面和提交粒度。

## 2026-03-12 v0.16

### 本次调整

1. 面试回合作答前新增 `intent router`，先判断输入属于 `answer/clarify/question_back/skip/meta/invalid` 哪一类。
2. 只有 `intent=answer` 才进入评分链路，避免把“反问面试官、问流程、要求解释题意”误判成可评分回答。
3. 评分从“规则层关键词/长度/证据数启发式”升级为“LLM rubric 评分为主，规则层仅作 fallback”。
4. rubric 固定四维：`technical_depth / structure_clarity / evidence_grounding / role_fit`。
5. 评分后新增第二步“最终评价生成”，后端把 rubric 结果转成自然语言评价文本，再以 SSE token 流式推给前端。
6. 前端不再先显示原始 JSON 再替换文案，只显示最终评价文本或非回答分支的面试官回复。
7. `skip` 分支会把当前题记为跳过并推进到下一题；`clarify/question_back/meta/invalid` 分支只回复，不计分、不推进。

### 当前链路

1. `classify intent`
2. `intent=answer` 时：
   - 检索简历/JD/用户资料
   - LLM rubric 评分
   - 生成自然语言评价
   - 写入 turn
   - 决定是否插入 follow_up 和下一题
3. `intent!=answer` 时：
   - 生成面试官回复
   - 视 intent 决定是否保留当前题或跳过当前题

### 这样做的原因

1. 之前所有输入都直接评分，更像“单次 LLM 调用”，不是一个有路由判断的面试 agent。
2. 关键词命中和证据数量只能做弱信号，不能直接代表回答质量。
3. 结构化 rubric 更适合内部存储和后续追问决策，但用户前台需要看到的是直接评价文本。

### 当前风险

1. 当前还是单 agent 多步骤编排，不是多 agent 协作；流程更清晰，但复杂场景的策略仍有限。
2. `intent router` 仍依赖 LLM 判断，后续可以继续叠加更细的规则校验和样本回放。
3. 非回答分支目前只做轻量回复，还没有接入更完整的会话策略树。

## 2026-03-13 v0.17

### 本次调整

1. 新增 `question_type router`，在 `intent=answer` 后继续识别 `basic/project/knowledge/scenario/follow_up`。
2. 面试评估链路的 Sirchmunk 检索改为显式走 `DEEP`，配合业务层先做题型路由和路径裁剪，优先提升项目题、场景题的证据召回质量。
3. 资料证据入口不再默认传整目录，而是按题型规划证据路径：
   - `basic`：活跃 JD
   - `project/scenario`：活跃 JD + 相关知识文档
   - `knowledge`：相关知识文档优先，必要时补充 JD
4. `resume` 不再作为证据源文件直接参与资料匹配，只保留 `resume_summary` 作为评分和标准答案生成的背景信息。
5. rubric 评分 prompt 新增 `question_type`，要求 LLM 区分项目题、知识题、场景题的判分重点。
6. 单题练习与模拟面试两条评分链路统一复用这套“题型路由 + 证据规划 + rubric”流程。
7. 用户文档目录物理拆分为 `profile` 与 `knowledge` 两层：
   - `profile`：仅存 `resume/jd`
   - `knowledge`：仅存检索知识库文档
   读取逻辑保留对旧平铺目录的兼容，避免历史数据立即失效。

### 当前链路

1. `intent router`
2. `question_type router`
3. `evidence planner`
4. `evidence collection`
5. `LLM rubric scoring`
6. `自然语言评价生成`

### 这样做的原因

1. 之前资料检索链路在意图分流阶段会把自我介绍类问题误判为整份资料 review。
2. 项目题不能只对齐简历，还要结合知识库判断方案是否合理，因此必须先做题型路由再规划证据源。
3. 评分应以“回答质量 + 证据支撑 + 项目/知识场景匹配”为主，而不是目录级总结或关键词命中。

### 当前风险

1. 题型识别仍是 LLM + fallback 混合判断，后续需要样本回放和命中统计。
2. `knowledge` 路由当前主要依赖文件名提示选知识文档，后续可升级为更细的文档索引。
3. WebSearch 仍是占位 fallback，尚未接入真实 provider。
