# 智能追问与面经召回方案

## 一、面经入库修改方案

### 1.1 chain vs mixed 区分

改造 cleaner prompt，区分真实追问链和同话题平行问题：

- **chain**：面试官针对同一回答逐步深入追问，`parent_ref` 指向实际追问的 item index
  - 判断标准：后一个问题明显是对前一个回答的深入/质疑/补充
  - 示例："说一下闭包" → "闭包会导致内存泄漏吗" → "怎么排查内存泄漏"
- **mixed**：同类但独立的问题，`parent_ref` 设为 null
  - 判断标准：同一个话题下的独立问题，面试官换了个角度问
  - 示例："说一下闭包"、"原型链怎么实现继承"、"Promise 有哪些状态"

### 1.2 chain_anchor 标注

区分通用追问链和简历特定追问链，**标注粒度为 item 级别**（而非 group 级别），因为实际场景中一条链可能前半段 generic、后半段 anchored（面试官从通用问题追到项目细节）：

- **generic**：追问基于知识点本身的递进，不依赖候选人具体经历
  - 信号：问题用"说一下/介绍/什么是/怎么理解"等通用提问方式
  - 入库方式：整条链原样入库
- **experience_anchored**：追问锚定在候选人的具体项目/经历上
  - 信号：问题中出现"你/你们/你这个项目/你当时/你们团队"等指代词
  - 入库方式：`question_text_raw` 保留原文，`question_text_normalized` 改写为去指代的通用形式
  - 示例：raw "你们项目里为什么选 Redis Cluster" → normalized "什么场景下应该选 Redis Cluster 而不是单机 Redis"

规则兜底（item 级别判断）：

```js
const EXPERIENCE_ANCHOR_SIGNALS = /你[们的]|你这个|你当时|你负责|你做的|你们团队|你项目|贵司|你公司/;

function inferItemChainAnchor(item) {
  return EXPERIENCE_ANCHOR_SIGNALS.test(item.question_text_raw)
    ? 'experience_anchored'
    : 'generic';
}
```

### 1.3 Embedding 入库

- 入库时对每个 group 算一次 embedding
- 输入：拼接 `canonical_question` + items 问题文本
- 存储：`experience_question_group.embedding_json` 列（JSON 数组）
- 服务启动时加载到内存 `Map<groupId, Float32Array>`
- 面经库规模 < 1 万条，内存暴力余弦搜索，不需要向量数据库

### 1.4 知识图谱增量更新

入库后同 group 内 `knowledge_points` 两两配对，对应边权重 +1：

```js
function updateGraphIncremental(graph, newGroups) {
  for (const group of newGroups) {
    const kps = [...new Set(group.items.flatMap(item => item.knowledge_points))];
    for (let i = 0; i < kps.length; i++) {
      const a = kps[i];
      if (!graph[a]) graph[a] = { parent: null, children: [], related: {}, count: 0 };
      graph[a].count++;
      for (let j = i + 1; j < kps.length; j++) {
        const b = kps[j];
        if (!graph[b]) graph[b] = { parent: null, children: [], related: {}, count: 0 };
        graph[a].related[b] = (graph[a].related[b] || 0) + 1;
        graph[b].related[a] = (graph[b].related[a] || 0) + 1;
      }
    }
  }
}
```

sync 批量完成后触发全量重建 + 孤立节点归位。

## 二、知识图谱建立方案

### 2.1 构建方式：LLM 骨架 + 面经库共现数据

**LLM 骨架（一次性生成，存为静态 JSON 文件）：**

- 输入 9 个一级分类：JavaScript、React、Vue、CSS、浏览器、工程化、性能优化、算法、Node.js
- **分批生成**：按一级分类各调一次 LLM，每次只生成一棵子树（50-80 个节点），最后合并。避免单次调用输出 500+ 节点导致遗漏或层级混乱
- LLM 输出每个节点的 parent、children、related
- 存为 `knowledge-graph-skeleton.json`

骨架示例：

```json
{
  "JavaScript": {
    "children": ["闭包", "原型链", "事件循环", "Promise", "this指向", "作用域", "类型判断", "ES6+"],
    "related": ["TypeScript", "Node.js"]
  },
  "闭包": {
    "parent": "JavaScript",
    "children": ["词法作用域", "内存泄漏", "柯里化", "IIFE"],
    "related": ["作用域", "垃圾回收", "this指向"]
  },
  "事件循环": {
    "parent": "JavaScript",
    "children": ["宏任务", "微任务", "requestAnimationFrame", "MessageChannel"],
    "related": ["Promise", "async/await", "Node.js事件循环", "浏览器渲染"]
  }
}
```

**面经库共现数据补充：**

- 遍历所有 group，同 group 内 knowledge_points 两两配对计数
- 共现次数 ≥ 2 的保留，过滤噪音（面经库初期数据量小时可降至 ≥ 1，后续随数据增长调高）
- 合并规则：骨架决定结构（parent/children），共现数据决定 related 权重

### 2.2 更新流程

```
单篇入库 → 增量更新 related 共现权重（纯计数，不动父子关系）
sync 完成 → 全量重建共现 + 孤立节点归位
服务启动 → 从 knowledge-graph-computed.json 加载到内存
```

- 父子关系由 LLM 骨架定义，是前端知识体系本身的结构，不随面经数据变化
- 面经数据只影响 related 的共现权重

### 2.3 孤立节点归位

面经库出现骨架中没有的新知识点时，创建无 parent 的孤立节点。归位策略：

1. **规则推断（优先）**：孤立节点共现最多的邻居节点，其 parent 大概率就是孤立节点的 parent

```js
function inferParentFromRelated(graph, orphanName) {
  const node = graph[orphanName];
  const topRelated = Object.entries(node.related)
    .sort((a, b) => b[1] - a[1]);
  for (const [relatedName] of topRelated) {
    const relatedNode = graph[relatedName];
    if (relatedNode?.parent) return relatedNode.parent;
  }
  return null;
}
```

2. **LLM 批量归位（兜底）**：推断不了的孤立节点攒起来，sync 完成后一次 LLM 调用批量处理

## 三、查询修改方案（召回）

### 3.1 简历结构化提取

改造现有 `summarizeResumeWithLLM`，从 120 字摘要改为结构化 JSON 输出：

```json
{
  "summary": "3年前端经验，主导电商平台缓存重构和微前端改造",
  "projects": [
    {
      "name": "电商平台缓存重构",
      "original_description": "负责电商平台缓存层重构，使用 Redis Cluster 替换本地 LRU 缓存，接入 CDN 回源策略优化，将首屏 LCP 从 3.2s 降至 1.1s，缓存命中率从 60% 提升至 92%",
      "tech_entities": ["Redis Cluster", "CDN", "LRU"],
      "key_features": ["Redis Cluster 替换本地 LRU 实现分布式缓存", "CDN 回源策略优化降低源站压力", "首屏 LCP 从 3.2s 优化至 1.1s"]
    }
  ],
  "all_tech_entities": ["Redis Cluster", "CDN", "React 18", "Webpack 5", "Module Federation"]
}
```

- LLM 自由提取 tech_entities，不塞词表，不做规则映射
- 同一次 LLM 调用既拿到摘要（给出题 LLM 用）又拿到结构化数据（给召回用）
- original_description 完整保留简历原文中该项目的描述段落，不归纳不压缩，后续 embedding 召回直接用原文
- key_features 提取 2-5 个关键技术实现点，用于知识图谱展开和精排

### 3.2 检索流程：知识图谱展开 + embedding 召回 + 图谱重排

```
简历项目描述
      ↓
LLM 提取 tech_entities
      ↓
知识图谱展开 (expandWithGraph)
      ↓
  ┌─ 增强 embedding query → 余弦召回 top 20
  │
  └─ 展开词集 → 对召回结果 rerank (rerankWithGraph)
      ↓
  应用层精排 (难度匹配 + 薄弱点 + 贪心去重)
      ↓
  最终 2-3 条 chain
```

### 3.3 知识图谱展开

从简历 tech_entities 出发，沿知识图谱展开一层，带权重衰减：

```js
function expandWithGraph(techEntities, graph, { maxDepth = 1, maxPerNode = 5, minWeight = 3 }) {
  const expanded = new Map();
  for (const entity of techEntities) {
    expanded.set(entity, 10); // 原始词最高权重
  }

  let currentLayer = [...techEntities];
  for (let depth = 0; depth < maxDepth; depth++) {
    const nextLayer = [];
    const decay = 1 / (depth + 2);

    for (const term of currentLayer) {
      const node = graph[term];
      if (!node) continue;
      const neighbors = Object.entries(node.related)
        .filter(([_, weight]) => weight >= minWeight)
        .sort((a, b) => b[1] - a[1])
        .slice(0, maxPerNode);

      for (const [neighbor, coWeight] of neighbors) {
        const score = coWeight * decay;
        if (!expanded.has(neighbor) || expanded.get(neighbor) < score) {
          expanded.set(neighbor, score);
          nextLayer.push(neighbor);
        }
      }
    }
    currentLayer = nextLayer;
  }
  return expanded;
}
```

### 3.4 增强 embedding query

把展开后的高权重词拼到项目描述后面，再算 embedding：

```js
function buildEnhancedQuery(projectDesc, expandedTerms) {
  const topTerms = [...expandedTerms.entries()]
    .filter(([term]) => !projectDesc.includes(term))
    .sort((a, b) => b[1] - a[1])
    .slice(0, 5)
    .map(([term]) => term);

  return topTerms.length > 0
    ? `${projectDesc}。相关: ${topTerms.join(' ')}`
    : projectDesc;
}
```

### 3.5 图谱重排

embedding 返回 top 20 后，用展开词集给结果加分：

```js
function rerankWithGraph(embeddingHits, expandedTerms) {
  const maxGraphBonus = Math.max(...embeddingHits.map(hit => {
    const kps = hit.chain_items.flatMap(item => item.knowledge_points);
    return kps.reduce((sum, kp) => sum + (expandedTerms.get(kp) || 0), 0);
  }), 1);

  return embeddingHits.map(hit => {
    const kps = hit.chain_items.flatMap(item => item.knowledge_points);
    const rawGraphBonus = kps.reduce((sum, kp) => sum + (expandedTerms.get(kp) || 0), 0);
    // 归一化到 0~10，与 cosineSimilarity * 10 量纲对齐
    const normalizedGraphBonus = (rawGraphBonus / maxGraphBonus) * 10;
    return { ...hit, finalScore: hit.cosineSimilarity * 10 + normalizedGraphBonus };
  }).sort((a, b) => b.finalScore - a.finalScore);
}
```

### 3.6 应用层精排 + 贪心去重

多维打分后贪心选取，每选一条标记已覆盖知识点，防止问题扎堆：

```js
function rankRecalledChains({ chains, candidateProfile, limit = 2 }) {
  const { knowledgeTags, targetLevel, weaknessTags = [], askedKnowledgePoints = [] } = candidateProfile;

  const DIFFICULTY_FIT = {
    junior: { easy: 3, medium: 2, hard: 0 },
    mid:    { easy: 1, medium: 3, hard: 2 },
    senior: { easy: 0, medium: 2, hard: 3 },
  };

  const scored = chains.map(chain => {
    const chainKPs = chain.chain_items.flatMap(item => item.knowledge_points);
    const uniqueKPs = [...new Set(chainKPs)];

    const tagScore = uniqueKPs.filter(kp => knowledgeTags.includes(kp)).length * 3;
    const mainDifficulty = chain.chain_items[0]?.difficulty || 'medium';
    const difficultyScore = DIFFICULTY_FIT[targetLevel]?.[mainDifficulty] ?? 1;
    const weaknessScore = uniqueKPs.filter(kp => weaknessTags.includes(kp)).length * 5;
    const qualityScore = (chain.frequency_score || 0) * 2;
    const overlapPenalty = uniqueKPs.filter(kp => askedKnowledgePoints.includes(kp)).length * -4;
    const chainBonus = chain.group_type === 'chain' ? 3 : 0;

    return {
      ...chain,
      _rankScore: tagScore + difficultyScore + weaknessScore + qualityScore + overlapPenalty + chainBonus,
      _knowledgePoints: uniqueKPs,
    };
  });

  scored.sort((a, b) => b._rankScore - a._rankScore);

  const selected = [];
  const coveredKPs = new Set(askedKnowledgePoints);
  for (const chain of scored) {
    if (selected.length >= limit) break;
    const overlap = chain._knowledgePoints.filter(kp => coveredKPs.has(kp)).length;
    if (chain._knowledgePoints.length > 0 && overlap / chain._knowledgePoints.length > 0.7) continue;
    selected.push(chain);
    chain._knowledgePoints.forEach(kp => coveredKPs.add(kp));
  }
  return selected;
}
```

## 四、面经调用方案（面试过程中）

### 4.1 出题阶段

- LLM 生成 3 题 + 面经库召回 2 条 chain
- 面经库的题带完整 follow_up chain，存到 `_follow_up_chain` 字段
- 合并进 interview_question 队列，`source_ref` 改为 group 级别引用（`experience_group:{groupId}`）

### 4.2 追问阶段

```
用户答完 → 评分 → 输出 knowledge_boundary
  ├─ mentioned_but_shallow（提到但浅）
  ├─ conspicuously_absent（明显遗漏）
  └─ potential_misconception（可能的错误认知）
            ↓
当前问题有 _follow_up_chain？
  ├─ 有 → rankFollowUpByRelevance 按 knowledge_boundary 选最相关的一条
  │       → adaptFollowUp 轻量 LLM 改写（~150 tokens）
  │       → chain_anchor = experience_anchored 时结合候选人简历重新锚定
  │
  └─ 没有 → 拿候选人回答原文 BM25 搜面经库相关追问链
            ├─ 搜到 → 同上 rank + adapt
            └─ 没搜到 → generateFollowUpQuestion（现有逻辑）
                        → 生成后回写面经库，创建新 chain group
```

### 4.3 追问选择排序

```js
function rankFollowUpByRelevance(chainItems, knowledgeBoundary) {
  // 优先级：
  // 1. knowledge_points 命中 potential_misconception → 纠正错误认知
  // 2. knowledge_points 命中 mentioned_but_shallow → 深挖浅层回答
  // 3. knowledge_points 命中 conspicuously_absent → 补盲区
  // 4. follow_up_intent: deepen > compare > verify > clarify
  return chainItems
    .map(item => ({ ...item, score: calcRelevanceScore(item.knowledge_points, knowledgeBoundary) }))
    .sort((a, b) => b.score - a.score)[0];
}
```

### 4.4 追问适配

```js
async function adaptFollowUp(bankItem, sessionContext) {
  const instruction = bankItem.chain_anchor === 'experience_anchored'
    ? '这道题原本针对特定项目经历。请结合候选人简历，把通用问题重新锚定到候选人的具体项目上。'
    : '把题库问题改写为口语化追问，贴合当前对话上下文。';

  return await llmCall({
    system: `你是面试官。${instruction}只改措辞，不改考察点。一句话输出。`,
    user: JSON.stringify({
      original_question: bankItem.question_text_normalized,
      knowledge_points: bankItem.knowledge_points,
      candidate_resume_project: sessionContext.currentResumeAnchor,
      recent_answer_excerpt: sessionContext.lastAnswer?.slice(0, 300)
    }),
    max_tokens: 150
  });
}
```

### 4.5 回写机制

LLM 生成的追问**异步**回写到面经库（不阻塞面试响应），标记 `source: "llm_generated"` 区分人工面经，避免污染召回质量：

```js
async function saveToExperienceBank(parentQuestion, generatedFollowUp) {
  // 异步执行，不阻塞面试流程
  setImmediate(async () => {
    let groupId = parentQuestion.source === 'experience'
      ? await getGroupIdFromSourceRef(parentQuestion.source_ref)
      : await createNewGroup({
          topic_cluster: parentQuestion.category,
          canonical_question: parentQuestion.stem,
          group_type: 'chain'
        });

    await insertExperienceQuestionItem({
      group_id: groupId,
      question_text_raw: generatedFollowUp.stem,
      question_text_normalized: generatedFollowUp.stem,
      question_role: 'follow_up',
      source: 'llm_generated', // 区分人工面经和 LLM 生成
      parent_item_id: parentQuestion.source_ref?.split(':')[1] || null,
      category: parentQuestion.category,
      difficulty: generatedFollowUp.difficulty,
      follow_up_intent: generatedFollowUp.follow_up_intent || 'deepen',
      knowledge_points_json: JSON.stringify(generatedFollowUp.knowledge_points || []),
      expected_points_json: JSON.stringify(generatedFollowUp.expected_points || [])
    });

    // 回写后增量更新知识图谱 + 重算 group embedding（同样异步）
  });
}
```

## 六、技术选型说明

### 6.1 为什么用 embedding 而不是标签交集/BM25

- **标签交集**：依赖两端标注同一个标准词，需要维护词表 + 同义词映射，每层规则映射都有信息损失
- **BM25**：适合自由文本匹配，但简历是陈述句、面经是疑问句，词汇空间天然不重合
- **embedding**：直接用自然语言原文算向量，零规则、零词表、零信息损失

BM25 适合的场景：面试过程中拿候选人回答原文搜面经库追问链（回答和问题词汇空间重叠）。

### 6.2 为什么不需要向量数据库

面经库规模 < 1 万条 group，embedding 维度 1536，内存占用约 30MB。暴力余弦遍历耗时 < 10ms。

```
< 1万条:   内存暴力搜索，零依赖
1万-10万:  pgvector 或 SQLite-vss
> 10万:    专用向量数据库 (Qdrant/Milvus)
```

### 6.3 知识图谱的作用

embedding 只能匹配语义相近的文本，匹配不了"知识图谱上相邻但表述不相关"的内容。

- embedding 能匹配："LCP 优化" ↔ "怎么优化首屏加载速度"
- embedding 匹配不到："LCP" ↔ "FCP 和 CLS 的区别"（同属性能指标但语义距离远）

知识图谱补充这层关联：从 embedding 命中的知识点出发，沿图谱展开相邻节点，增强 query 并参与重排。
